# AI编程统一范式规划草案

## 1. 目标与适用范围

- 目标：建立一套跨语言、跨技术栈、面向多Agent协作的统一范式，约束从需求到复盘的全过程。
- 适用对象：
  - 架构师
  - 编码者
  - 测试者
- 适用场景：新功能、缺陷修复、重构、性能优化、稳定性治理、技术债清理。
- 非目标：不绑定具体框架与工具，不限定单一开发模型。

---

## 2. 治理原则

1. 单一事实源
   - 每个阶段必须有可追溯产物，禁止口头状态驱动执行。
2. 先定义再实现
   - 未完成输入契约与验收标准，不进入编码。
3. 证据优先
   - 结论必须绑定证据：日志、测试结果、差异说明、门禁审查记录。
4. 最小变更面
   - 优先最小可验证改动，降低回归风险。
5. 门禁驱动
   - 阶段转换由质量门禁决定，而非角色主观判断。
6. 异常显式化
   - 风险、阻塞、假设必须显式记录与升级。
7. 跟踪闭环
   - 需求跟踪表必须跨阶段持续更新，并作为门禁与交接的共同依据。

## 3. 角色模型与职责边界

### 3.1 架构师
- 负责：将客户需求细化为单一文档并驱动后续工作；同时承担问题定义、目标分解、范围边界、优先级与验收标准草拟、总体设计、单元设计、接口设计、技术取舍、风险评估、演进路径；并将编码与测试工作细化为可分发、可验收的执行单元包。
- 输出：单一需求设计主文档、总体设计说明、单元设计说明、接口定义清单、成功标准、约束清单、技术决策记录、风险处置方案、执行单元包清单、需求跟踪表。
- 交付核验要点：统一需求主文档标识、总体设计说明、单元设计说明、接口定义清单、执行单元包清单、编码与测试单元包映射、约束与风险处置完整、需求跟踪表可追溯。
- 禁止：在关键需求或关键设计未决时推动实现。

### 3.2 编码者
- 负责：按设计实现、补充测试、维护可读性与可维护性。
- 输出：实现清单、实现阶段文档、需求跟踪表编码状态更新。
- 核查要点：
  - 执行单元包编号、需求编号映射、变更点映射、自测结果、待测试移交项完整。
  - 实现阶段文档必须包含日期、备注、风险、遗留事项、进度状态、完成情况、检查表、跟踪表状态、结论记录。
  - 需求跟踪表必须更新编码状态、当前责任角色、最新更新时间，且与实现清单一致。
- 核查失败处理：若编码核查未通过，必须在同一工作周期内补齐缺失字段并回写需求跟踪表，同时将状态标记为阻塞，禁止流转到测试阶段。
- 禁止：擅自偏离已批准的关键契约。

### 3.3 测试者
- 负责：测试设计与执行、问题复现、修复验证、回归风险评估。
- 输出：测试记录、测试阶段文档、需求跟踪表测试状态更新。
- 核查要点：
  - 测试单元包编号、需求编号映射、测试记录、复现证据、回归结论、未通过项处置状态完整。
  - 测试阶段文档必须包含日期、备注、风险、遗留事项、进度状态、完成情况、检查表、跟踪表状态、结论记录。
  - 需求跟踪表必须更新测试状态、未通过项处置状态、最新更新时间，且与测试记录一致。
- 核查失败处理：若测试核查未通过，必须记录失败项与处置计划并回写需求跟踪表，同时将状态标记为待修复，禁止进入复盘阶段。
- 禁止：无复现证据直接判定问题结论。

### 3.4 统一协作契约
- 每个角色接手任务前，必须确认：
  - 输入是否完整
  - 前置门禁是否通过
  - 目标与验收是否可执行
- 每个角色交接任务时，必须提供：
  - 输出产物链接
  - 未决问题
  - 风险与建议
  - 跟踪表状态
- 编码者与测试者交接文档必须显式包含日期、备注、风险、遗留事项、进度状态、完成情况、检查表、结论记录。
- 交接文档必须同时满足通用最小内容与角色扩展内容，并显式落盘到`/doc/`下对应角色目录。

## 4. 端到端流程与阶段I O定义

### 阶段S1 需求澄清
- 输入：问题背景、业务目标、约束条件、现状证据。
- 最小启动输入：用户需求原文、关键选型偏好。
- 输出：需求模板成稿、范围清单、验收标准、需求跟踪表初版。
- 完成标准：需求可验证、边界清晰、冲突项已记录；最小启动输入中的缺失信息已通过澄清问题补齐。

### 阶段S2 方案设计
- 输入：S1输出。
- 输出：设计模板成稿、总体设计说明、单元设计说明、接口定义清单、架构决策、风险清单、执行单元包清单、需求跟踪表更新版。
- 完成标准：关键路径可行、接口契约明确、风险有应对；编码与测试任务均已拆分为可执行、可验收单元。

### 阶段S3 实现与自测
- 输入：S2输出。
- 输出：实现清单、代码变更、测试证据、需求跟踪表编码状态更新、编码阶段文档。
- 完成标准：实现与设计一致，测试覆盖关键路径；编码阶段文档已完整记录日期、备注、风险、遗留事项、进度状态、完成情况、检查表、结论记录。

### 阶段S4 测试与修复验证
- 输入：缺陷现象、日志与监控、S3产物。
- 输出：测试记录、问题分析、修复与验证证据、需求跟踪表测试状态更新、测试阶段文档。
- 完成标准：问题闭环、回归验证通过；测试阶段文档已完整记录日期、备注、风险、遗留事项、进度状态、完成情况、检查表、结论记录。

### 阶段S5 复盘与规范回灌
- 输入：S1 S4阶段产物、门禁检查结果、异常记录、需求跟踪表最终状态。
- 输出：复盘模板、改进项、规范更新建议。
- 完成标准：形成可执行改进动作并入库。

### 最小输入启动路径
- 适用条件：用户仅提供需求与关键选型。
- 启动方式：由架构师在S1创建需求跟踪表初版并发起澄清清单，补齐目标、约束、验收与风险字段。
- 流转条件：澄清清单已关闭或转为已记录遗留后，进入S2设计与执行单元包拆分。

```mermaid
flowchart LR
A[S1 需求澄清] --> B[S2 方案设计]
B --> C[S3 实现与自测]
C --> D[S4 测试与修复验证]
D --> E[S5 复盘与规范回灌]
E --> B
```

## 5. 质量门禁与异常处理

## 5.1 规则分级

### 必须项
- 必须存在阶段输入输出产物。
- 必须存在可验证验收标准。
- 必须保留测试与验证证据。
- 必须记录未决项及其处置状态。
- 必须有异常回退策略。
- 必须维护需求跟踪表，并在S1到S5逐阶段更新状态与责任归属。
- 必须执行角色输出核验与文档交付边界核验，统一要求如下：
  - 阶段产物必须显式落盘到`/doc/`下对应角色目录，CLI对话不可作为唯一交付。
  - 每份阶段文档必须同时满足通用最小内容与角色扩展内容，且文档不包含代码内容。
  - 通用最小内容：日期、备注、风险、遗留事项、进度状态、完成情况、检查表、跟踪表状态与结论记录。
  - 角色扩展内容：
    - 架构师：统一需求主文档标识、总体设计说明、单元设计说明、接口定义清单、执行单元包清单、编码与测试单元包映射、约束与风险处置。
    - 编码者：执行单元包编号、变更点映射、自测结果、待测试移交项。
    - 测试者：测试单元包编号、测试记录、复现证据、回归结论、未通过项处置状态。

### 建议项
- 建议保留替代方案与取舍理由。
- 建议沉淀可复用脚手架与检查脚本。
- 建议维护缺陷模式库与预防策略。

## 5.2 门禁检查点
- G1 需求门：架构师输出的目标、范围、验收完整，需求跟踪表初版已建立。
- G2 设计门：架构师输出的总体设计、单元设计、接口定义、风险、执行单元包拆分完整，需求跟踪表已完成设计映射。
- G3 编码核查门：编码者输出的实现一致性、自测结果、映射关系完整；编码阶段文档通用字段齐全；需求跟踪表编码状态、责任角色、更新时间与实现清单一致。
- G4 测试核查门：测试者输出的问题结论、回归结果、未通过项处置状态完整；测试阶段文档通用字段齐全；需求跟踪表测试状态、处置状态、更新时间与测试记录一致。
- G5 复盘门：跨角色输出闭环与改进项登记完整，需求跟踪表状态闭环。

## 5.3 异常分级与升级路径
- P0：核心功能失效或任务阻断，立即升级到架构师并触发跨角色应急协同。
- P1：关键功能退化，需限时修复并重新走门禁。
- P2：非关键缺陷，进入迭代修复并跟踪复盘。
- 所有异常必须包含：现象、影响范围、临时缓解、根因状态、下一步动作。

## 7. 跨角色执行机制

1. 检查清单机制
   - 每阶段提交时附带Checklist，必须项全部满足才可流转。
   - Checklist必须包含对应角色输出核验项，未满足时禁止流转。
   - Checklist必须包含需求跟踪表核验项，包括编号连续性、状态一致性、责任角色一致性、更新时间有效性。
   - 必须分别执行编码核查Checklist与测试核查Checklist，且两者均需留痕归档。
2. 审查节奏
   - 关键节点审查：需求冻结前、设计冻结前、实现冻结前、复盘后。
3. 升级路径
   - 角色内无法消解的阻塞，按P0 P1 P2路径升级。
4. 留痕要求
   - 决策、变更、验证、异常统一归档，支持追溯与审计。
   - 每阶段必须输出结构化文档到`/doc/角色目录/`，并通过链接在交接单中引用。
5. 规范回灌
   - 每次复盘产出的改进项进入规范候选，复核后纳入主规范版本。
   - 执行单元包拆分质量需纳入复盘结论，用于持续优化后续任务分发颗粒度。

## 9. 实施落地步骤

1. 冻结主规范目录与角色定义。
2. 生成六类模板初版 含需求跟踪表。
3. 建立需求跟踪表主文件并定义跨阶段更新规则。
4. 建立`/doc/角色目录/`输出结构与命名规则。
5. 为每阶段配置门禁清单。
6. 在一次真实任务中试运行并记录偏差。
7. 复盘后修订主规范与模板。
8. 固化为后续AI协作默认协议。

## 10. 文档落盘与目录规范

### 10.1 目录结构

```text
/doc/
  /architect/
  /coder/
  /tester/
```

- 所有阶段输出必须写入上述目录之一。
- 不允许仅在CLI中给出结论而不形成文档文件。
- 本目录用于过程治理文档，不用于承载代码实现内容。

### 10.2 文件命名规范

- 建议命名：`YYYY-MM-DD_阶段_主题_角色.md`
- 示例：`2026-04-11_S2-设计_统一范式目录规范_architect.md`
- 需求跟踪表建议命名：`YYYY-MM-DD_需求跟踪表_版本号.md`

### 10.3 文档最小必填内容 含角色扩展

每份阶段文档必须至少包含以下内容组：

- 通用内容组
  - 基本标识：日期、阶段、角色、任务主题。
  - 治理信息：进度状态、完成情况、检查表、跟踪表状态、结论记录。
  - 风险与遗留：风险、遗留事项、未决项处置状态。
  - 关联追溯：关联文档链接。

- 角色扩展内容组
  - 架构师文档：统一需求主文档标识、需求跟踪表、总体设计说明、单元设计说明、接口定义清单、执行单元包清单、编码与测试单元包映射、约束与风险处置。
  - 编码者文档：执行单元包编号、需求编号映射、变更点映射、自测结果、待测试移交项。
  - 测试者文档：测试单元包编号、需求编号映射、测试记录、复现证据、回归结论、未通过项处置状态。

### 10.4 交接强制要求

- 交接时必须引用文档路径，如`/doc/architect/2026-04-11_S2-设计_xxx_architect.md`。
- 若缺少`10.3`定义的任一最小内容组，门禁判定不通过。
- S5复盘阶段需核验文档存在性、字段完整性、链接可追溯性。

### 10.5 文档内容边界

- 文档内容边界以`10.3`最小必填内容为准，仅记录治理信息。
- 文档不直接承载代码片段、补丁或可执行实现细节。
- 若需引用实现信息，仅允许以外部链接形式指向代码仓库提交或变更记录。
