# 混合编码通用方案 仅保留安全写入与 CI 门禁

## 1. 目标
在 UTF-8 终端环境下，确保对历史 GBK C C++ 文件执行增删改时不发生误转码，并在 CI 阶段自动阻断不合规编码提交。

## 2. 范围
仅保留两项能力：
1. 安全写入脚本
2. CI 编码门禁脚本

不包含本地编辑器设置、不包含 pre-commit 说明、不包含 Git 其他策略。

## 3. 交付物
1. 安全写入脚本 `tools/encoding_safe_patch.py`
   - 支持操作：replace insert_before insert_after delete
   - 支持模式：dry-run 与 apply
   - 按扩展名映射编码
   - 写入前自动备份，失败可回滚
   - 输出 JSON 报告
2. CI 门禁脚本 `tools/encoding_ci_gate.py`
   - 扫描指定文件集合
   - 按编码映射逐文件校验解码
   - 校验失败返回非零退出码
   - 输出失败文件清单与原因
3. 编码策略文件 `tools/encoding-policy.json`
   - 维护扩展名或 glob 到 encoding 的映射

## 4. 脚本设计
### 4.1 安全写入脚本
输入
- patch 配置文件 UTF-8 JSON
- 每个变更项包含
  - path
  - encoding 可选，缺省按 policy 推导
  - operations

流程
1. 读取目标文件原始字节
2. 按目标编码解码，失败即中断
3. 执行文本级增删改
4. 校验命中数与期望一致
5. dry-run 输出报告不落盘
6. apply 先备份再写回原编码字节

输出
- 执行报告 JSON
- 备份目录

### 4.2 CI 门禁脚本
输入
- policy 文件路径
- 待扫描目录与文件模式

流程
1. 收集待检查文件
2. 根据 policy 推导期望编码
3. 逐文件按期望编码解码验证
4. 记录失败项
5. 失败则退出码 1，成功退出码 0

输出
- 控制台摘要
- 可选 JSON 报告

## 5. 编码策略建议
- .c .cpp .h .hpp -> gbk
- .py .md .json .yml .yaml .cmake -> utf-8

## 6. 最小实施顺序
1. 新建 `tools/encoding-policy.json`
2. 实现 `tools/encoding_safe_patch.py`
3. 实现 `tools/encoding_ci_gate.py`
4. 在 CI 中增加门禁步骤调用 `tools/encoding_ci_gate.py`

## 7. 验收标准
1. 在 UTF-8 终端下执行安全写入脚本修改 GBK 文件后无乱码
2. 安全写入脚本 dry-run 与 apply 输出一致的命中统计
3. CI 门禁能准确阻断编码不合规文件
4. 安全写入脚本可基于备份恢复原始字节
