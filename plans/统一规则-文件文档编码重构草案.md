# 统一规则重构草案 文件与文档编码规则

## 1. 重构依据
- 方案依据：`plans/混合编码通用治理方案.md`
- 工具依据：`tools/encoding-policy.json`、`tools/encoding_safe_patch.py`、`tools/encoding_ci_gate.py`

## 2. 重构目标
将原统一规则中“程序源代码字符集编码”相关条款重构为“文件与文档编码统一规则”，并严格对齐当前可执行能力：
1. 安全写入脚本
2. CI 编码门禁脚本

## 3. 新规则核心口径

### 3.1 编码基线以 policy 为唯一来源
- 仓库编码基线由 `tools/encoding-policy.json` 定义并维护。
- 默认编码由 `default_encoding` 决定。
- 文件级编码由 `rules` 中的 glob 匹配优先决定。
- 编码范围仅按 policy 解释，不再要求角色口头询问字符集策略。

### 3.2 文件与文档编码分类
- 历史 C C++ 相关文件（如 `.c .cpp .h .hpp .inl`）按 policy 使用 `gbk`。
- 文档与脚本类文件（如 `.md .txt .json .yml .yaml .py .cmake CMakeLists.txt`）按 policy 使用 `utf-8`。
- 若新增扩展名或路径模式，必须先更新 policy，再进行实现与提测。

### 3.3 混合编码文件修改方式
- 对受混合编码治理影响的文件修改，统一使用 `tools/encoding_safe_patch.py` 执行。
- 脚本执行流程必须满足：按目标编码解码、执行文本操作、命中数校验、apply 前备份、输出 JSON 报告。
- 禁止在规则中引入超出脚本能力范围的硬性要求。

### 3.4 CI 门禁
- 编码合规统一由 `tools/encoding_ci_gate.py` 执行。
- 门禁按 policy 推导期望编码并逐文件解码校验。
- 任一文件解码失败即门禁不通过并阻断流转。

### 3.5 非范围声明
- 统一规则编码章节不包含本地编辑器设置。
- 不包含 pre-commit 约束。
- 不包含 Git 其他编码策略。

## 4. 对原统一规则的定向重构建议

### 4.1 全局强制规则补强项
- 在“文档必须显式落盘”附近新增：阶段文档属于 policy 中的文档类型，按 policy 指定编码保存。
- 在“文档边界”附近新增：仅约束可执行脚本与门禁口径，不引入编辑器与提交钩子策略。

### 4.2 角色规则重构项
- 架构师规则：
  - 删除“首次承接需求必须显式询问字符集编码要求”。
  - 改为“确认本次任务编码基线来源为 policy；如需变更，先更新 policy 并记录变更原因”。
- 编码者规则：
  - 将“按架构师指定字符集编码实施”改为“按 policy 解析后的编码实施”。
  - 涉及混合编码文件时，要求使用 `encoding_safe_patch.py`。
- 测试者规则：
  - 增加“执行或核验 CI 编码门禁结果”。

### 4.3 门禁规则重构项
- G2 架构门：
  - 删除“首次显式询问字符集编码”核验。
  - 改为核验“policy 基线已确认，若有变更已记录并可追溯”。
- G3 编码核查门：
  - 保留“实现文件与编码基线一致”。
  - 明确“一致性判定证据来自安全写入报告或等效可追溯证据”。
- G4 或 G6：
  - 增加“`encoding_ci_gate.py` 执行通过”作为阶段放行前置条件。

### 4.4 章节替换建议
- 将原“程序源代码字符集编码章节”整体替换为“文件与文档编码统一规则”章节，结构建议：
  1. 编码基线来源与优先级
  2. 文件分类与默认映射
  3. 安全写入执行规范
  4. CI 门禁执行规范
  5. 变更控制与审计证据
  6. 非范围声明

## 5. 与工具行为一致性对照
- 与 `tools/encoding-policy.json` 一致：基于 glob 映射与 default fallback。
- 与 `tools/encoding_safe_patch.py` 一致：支持 replace insert_before insert_after delete、dry-run/apply、备份与报告。
- 与 `tools/encoding_ci_gate.py` 一致：按策略解码验证、失败非零退出码、可输出报告。

## 6. 建议实施顺序
1. 先改统一规则中的编码章节与门禁条款。
2. 再校正文档模板中的编码表述。
3. 最后做一次“规则条款 vs 工具行为”一致性复核。
